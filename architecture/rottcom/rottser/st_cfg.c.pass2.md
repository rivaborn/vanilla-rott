## Design Patterns & Rationale

**Environment-driven file discovery**: GetPathFromEnvironment() combines DOS environment variables with fallback behaviorâ€”if APOGEECD is unset, use the filename directly. This pattern reflects how Apogee distributed games on CD media where the drive path was unknown at install time.

**Sequential token validation**: CheckParameter() enforces strict token order in config files. This is fragile but avoids the complexity of a proper parser; missing or out-of-order tokens cause immediate Error() termination (which halts the program). The design assumes config files are generated by SETUP.EXE and won't be hand-edited.

**Sentinel-based optional parameters**: Tilde ("~") marks unconfigured parameters. For modem init/hangup strings, if `~` is found and usemodem is true, an error is thrown; otherwise defaults to "ATZ" (modem reset command). For serial port settings, `~` means -1 (disabled). This is a lightweight approach to optional config without explicit boolean flags.

**Script-based configuration**: Rather than struct-based binary configs or ini files, ROTT uses tokenized text scripts. This leverages scriplib's existing parsing infrastructure and allows human-readable (though not human-editable) config.

## Learning Notes

**90s game configuration idiom**: This reflects an era when configuration was file-based and environment-dependent, rather than registry/JSON/embedded. The reliance on APOGEECD and strict token order suggests code generated by a proprietary setup tool rather than generic.

**Contrast to modern engines**: Today's engines use structured formats (JSON, YAML, Lua), support hot-reload, and validate schemas. ROTT's approach is brittleâ€”any change to SETUP.ROT format or token order breaks the parser.

**Serial/modem context**: This file is specific to the modem/serial gameplay features (IPX/serial multiplayer). Single-player or modern network code wouldn't execute this path, making it legacy infrastructure within ROTT.

## Potential Issues

- **Buffer overflow**: GetPathFromEnvironment() uses strcpy/strcat without bounds checking; a very long APOGEECD path could overflow `fullname` buffer
- **Rigid token order**: ReadSetup() assumes tokens appear in exact sequence; any deviation causes CheckParameter() to call Error(), which terminates the program
- **No value validation**: Parsed baud rates, COM port numbers, and UART addresses are not validated for reasonable ranges
- **Error messaging**: Calls to Error() are terminal; there's no graceful fallback if config is malformed
- **Memory leak risk**: If LoadScriptFile() fails or Error() is called mid-function, scriptbuffer is not freed (though Error() likely exits before this matters)
- **DOS-specific assumptions**: Code assumes DOS-style paths with backslashes; porting to other platforms requires path handling changes

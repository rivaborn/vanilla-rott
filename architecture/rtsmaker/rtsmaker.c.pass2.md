# rtsmaker/rtsmaker.c — Enhanced Analysis

## Architectural Role
RTSMAKER is a **build-time utility** (not a runtime engine component) that exists in the tool ecosystem alongside the game engine. It serves as a bridge between raw sound assets (VOC/WAV files) and the Duke Nukem 3D/Rise of the Triad multiplayer subsystem, which expects remote taunt sounds to be packaged in WAD containers. This utility operates completely independently from the game engine (rott/ folder), using only generic build-time libraries (cmdlib, scriplib) for file I/O and configuration parsing.

## Key Cross-References
### Incoming (who depends on this file)
- **Game engine build system**: The output RTS files generated by this utility are consumed at runtime by the game's audio/network subsystem (likely referenced in game code that loads remote taunt sounds)
- No runtime game code directly calls rtsmaker functions—this is a standalone executable

### Outgoing (what this file depends on)
- **cmdlib subsystem**: `SafeOpenRead()`, `SafeOpenWrite()`, `SafeRead()`, `SafeWrite()`, `SafeMalloc()`, `SafeFree()`, `DefaultPath()`, `DefaultExtension()`, `ExtractFileBase()`, `LittleLong()` for portable file I/O and memory management
- **scriplib subsystem**: `LoadScriptFile()`, `GetToken()` for script tokenization (global `token` buffer)
- **Standard C library**: `stdio.h`, `stdlib.h`, `string.h` for file operations and string manipulation
- **Win32 primitives**: Direct Win32 file handle APIs (`open`, `close`, `lseek`, `tell`) and path APIs (`getcwd`, `chdir`)

## Design Patterns & Rationale
**Streaming Chunked Copy** (`MAXCOPYSIZE = 16KB`): Rather than loading entire sound files into memory, `CopyLump()` reads and writes in 16KB chunks. This is economical for a build tool and consistent with 1990s tool design where memory was precious.

**WAD Container Format**: The entire format (lumpinfo_t, wadinfo_t) is inherited from the game engine's asset system. RTSMAKER treats sounds as "lumps" (asset chunks) with 8-character names and metadata pointers—a simple but rigid design that mirrors the main engine's architecture.

**Script-Driven Batch Operations**: Rather than command-line flags, rtsmaker reads a text script specifying output name and sound file list. This is typical of build tools from the Apogee era, allowing non-programmers to configure packaging.

**Label Boundaries** (`REMOSTRT`/`REMOSTOP`): The addition of zero-size labels at container boundaries suggests the runtime expects to scan for these markers when loading remote taunts, allowing the engine to skip unrelated lumps.

## Data Flow Through This File
```
Command-line args (script file or -u unpack)
    ↓
ProcessScript() [creation mode]
    ├→ CmdRTSName() — create output RTS, allocate lump table
    ├→ Loop: for each sound file
    │   ├→ OpenLump() — open input file
    │   └→ CopyLump() — chunk-read and stream-write to RTS
    └→ WriteDirectory() — write lump table + header, close
    
OR

UnpackRTS() [unpack mode]
    ├→ Read RTS header + lump table
    ├→ For each lump: detect format (first byte 'C'=.voc/.wav=.wav)
    └→ Extract to individual files on disk
```

## Learning Notes
**Idiomatic to This Era**: 
- Direct file handle management (open/close/lseek/tell) rather than buffered streams
- Fixed-size buffers and string arrays (MAXPATH, 8-char lump names)
- No error recovery—errors fatal via `Error()` macro
- Manual byte-order conversion (`LittleLong()`) for little-endian WAD format

**Modern Equivalent**: A build-time sound asset packer would likely use JSON config, dynamic memory, and proper error handling.

**Why WAD Format?**: The game engine already has WAD loading code; reusing it here avoids duplicating asset-reading logic and ensures the tool and engine stay in sync.

## Potential Issues
1. **Fixed Lump Limit** (`MAXSOUNDS = 10`): The utility enforces exactly 10 sounds per RTS; trying to package fewer or more than 10 will error. This is inflexible but matches the game's expected remote taunt slot count.

2. **Hardcoded IWAD ID**: WAD header always written as "IWAD", but no validation that the input script is for internal assets only (vs. shareware). Mixing asset sources could silently break WAD loading.

3. **Buffer Overflow Risk**: `AddLabel()` calls `strupr(token)` but should operate on the input `string` parameter (likely a bug in line where `token` is used instead of `string`).

4. **No Validation of Lump Names**: 8-character lump names are truncated without warning; collisions are possible and will silently overwrite in the lump table.

# audiolib/source/mpu401.c — Enhanced Analysis

## Architectural Role

This file implements the low-level hardware driver layer for MPU401 MIDI interfaces in the audio subsystem. It sits at the bottom of a three-tier MIDI stack: high-level game code calls `AL_*` functions (in `al_midi.c`), which dispatch to device-specific drivers like `mpu401.c`, `awe32.c`, or `adlibfx.c` depending on available hardware. The MPU401 driver provides the minimal, raw MIDI I/O abstraction—single-byte transmission with busy-wait polling—allowing the AL layer to compose and send MIDI events without knowing hardware details.

## Key Cross-References

### Incoming (who depends on this file)
- **`al_midi.c` (AL subsystem)**: High-level MIDI composition layer calls `MPU_NoteOn()`, `MPU_NoteOff()`, `MPU_ControlChange()`, etc. via function pointers or direct calls after device detection. The AL layer abstracts instrument/voice management and delegates raw MIDI transmission to this driver.
- **Initialization chain**: Game startup (`rt_main.c`-style code) calls `MPU_Init(addr)` during audio subsystem setup, which in turn queries `USER_GetText("MPUDELAY")` for tuning parameters.
- **Real-time audio context**: `MPU_LockMemory()` is invoked before playback to ensure MIDI transmission code runs without page faults (critical for DOS protected mode).

### Outgoing (what this file depends on)
- **`mpu401.h`**: Extern declarations of `MPU_BaseAddr`, `MPU_Delay`, and MIDI/MPU command constants (0xff for reset, 0x3f for UART mode, etc.).
- **`user.h`**: `USER_GetText()` retrieves the "MPUDELAY" configuration string; this allows users/developers to tune delay loops for slow MIDI interfaces.
- **`dpmi.h`**: `DPMI_LockMemoryRegion()`, `DPMI_Lock()`, etc., lock critical code into physical memory. This is DOS-specific and reflects the era's protected-mode concerns.
- **Standard I/O**: `inp()` and `outp()` from `<conio.h>` perform raw I/O port access (x86 IN/OUT instructions).

## Design Patterns & Rationale

**Polling-based Hardware Access**: Rather than interrupt-driven I/O (which would require an ISR and IRQ allocation), this driver uses busy-wait loops to poll the MPU401 status port. This is simpler to implement and avoids interrupt handler complexity but burns CPU cycles and blocks the main thread.

**Configurable Delay Loop**: The `MPU_Delay` variable (default 0x5000, configurable via "MPUDELAY") allows tuning for hardware variations. Some MPU401 clones or slow ISA cards needed longer waits before responding—hardcoding a single delay would fail on slow hardware.

**Memory Locking Markers**: The pattern `MPU_LockStart` (defined as `MPU_SendMidi`) and `MPU_LockEnd` (an empty function) demarcates the range of code that must be locked into RAM. This prevents page faults during time-critical MIDI I/O in a DOS protected-mode environment (when using DPMI, a page fault handler invokes disk I/O, causing unacceptable latency).

**Retry Logic in Initialization**: `MPU_Init()` retries `MPU_Reset()` up to 4 times before giving up. This accommodates flaky hardware detection or cold-start issues common in ISA card initialization.

**Stateless Event Functions**: High-level MIDI events (`MPU_NoteOn`, `MPU_ControlChange`, etc.) are thin wrappers—they compose a status byte and two data bytes, calling `MPU_SendMidi()` three times. This keeps each function small and delegable; the AL layer handles sequencing and timing.

## Data Flow Through This File

1. **Initialization phase** (startup):
   - Game code → `MPU_Init(0x330)` 
   - Reads "MPUDELAY" env var via `USER_GetText()`; sets globals `MPU_BaseAddr`, `MPU_Delay`
   - Retries `MPU_Reset()` up to 4 times → sends reset command via `MPU_SendCommand()` → polls for acknowledgment
   - On success, calls `MPU_EnterUART()` to switch from intelligent to UART (raw) mode
   - If all succeeds, caller invokes `MPU_LockMemory()` to lock MIDI code into RAM

2. **Runtime phase** (playback):
   - AL layer → calls `MPU_NoteOn(ch, key, vel)` (or similar)
   - Wrapper composes three MIDI bytes: `[0x90|ch, key, vel]` by calling `MPU_SendMidi()` three times
   - Each `MPU_SendMidi()` call:
     - Polls status port (`MPU_BaseAddr + 1`) with mask `MPU_ReadyToWrite` (0x40) up to `MPU_Delay` iterations
     - Once ready, writes byte to data port (`MPU_BaseAddr`)
   - Hardware queues or transmits the MIDI event to synthesizer

3. **Shutdown phase**:
   - Game shutdown → calls `MPU_UnlockMemory()` to release DPMI locks

## Learning Notes

**Era-specific Real-Time Constraints**: DOS protected mode (via DPMI) could incur page faults if code/data wasn't locked, causing millisecond-scale stalls. Modern systems have CPU caches, DMAs, and OS-managed paging that hide latency; this driver reflects 1990s bare-metal thinking where every cycle and every page fault mattered.

**Polling vs. Interrupts**: The busy-wait approach is CPU-intensive but avoids interrupt handler complexity. A modern driver would use event-driven I/O; this one trades CPU for simplicity and determinism (no IRQ contention).

**Hardware Compatibility**: The configurable delay loop and retry logic show pragmatism—MPU401 was a pseudo-standard interface, and hardware varied. The driver accommodates both fast and slow variants without compile-time configuration.

**Three-Tier Abstraction**: MPU401 sits beneath the AL (audio library) layer, which itself may sit beneath game-level audio code. This separation allows swapping MPU401 for AWE32, Gravis Ultrasound MIDI, or other devices without changing AL or game code—a common 1990s game engine pattern.

**Implicit Sequencing**: Higher-level code is responsible for MIDI note sequencing and timing; this driver only guarantees byte delivery with polling. There's no queue, no timestamp, no note-off scheduling—that's the AL layer's job.

## Potential Issues

- **No explicit timeout mechanism**: `MPU_Reset()`, `MPU_EnterUART()`, and `MPU_SendCommand()` use fixed iteration counts (0xffff or `MPU_Delay`) rather than wall-clock timeouts. On a very slow or unresponsive card, the polling loops burn CPU for a predictable but not wall-time-bounded duration.
- **Unvalidated user input**: `USER_GetText("MPUDELAY")` is converted via `atol()` with no range checking. An invalid or negative value could cause unexpected behavior.
- **No error recovery in `MPU_SendMidi()`**: Unlike `MPU_SendCommand()`, `MPU_SendMidi()` does not return a status code. If the device hangs after initialization, subsequent note-on calls silently fail to transmit.
- **CPU-blocking I/O**: Busy-wait loops block the entire thread. On multi-threaded or preemptive systems, this would be unacceptable; in DOS, it's acceptable because single-threaded execution is the norm.
